## 2. Relationship Model

```
┌──────────────────────────────────────────────────────┐
│              Salesforce Org                          │
│                                                      │
│  EnrichmentScheduler (nightly cron)                  │
│        │                                             │
│        ▼                                             │
│  EnrichmentBatchProcessor                            │
│        │ calls                                       │
│        ▼                                             │
│  EnrichmentCalloutService ──POST──► /api/v1/enrich   │
│                                           │          │
│  EnrichmentWriteBack ◄─── Apex path       │          │
│  (inside Salesforce TX)                   │          │
│                                           │          │
│  ◄──────────────── SalesforceClient ◄─────┘          │
│  (Python OAuth2 path, independent TX)    Python API  │
│                                           │          │
│  Enrichment_Run__c (provenance) ◄─────────┘          │
└──────────────────────────────────────────────────────┘
```

**Ingress direction (Salesforce → Python):**
`EnrichmentCalloutService.enrich()` → `POST /api/v1/enrich` → `handle_enrich(tenant, payload)` → `EnrichmentOrchestrator` → KB resolver → consensus engine → response

**Write-back direction (dual paths — risk):**
- **Path A (Apex):** `EnrichmentWriteBack.apply()` — runs inside the Salesforce batch transaction, FLS-checked, synchronous DML
- **Path B (Python):** `app/services/crm/salesforce_client.py` + `app/services/crm/writeback.py` — calls Salesforce REST API from Python via OAuth2 session, independent transaction

**Idempotency surface:**
- `Enrichment_Run__c.Run_UUID__c` — Apex-side dedup key per enrichment attempt
- Python side: no shared dedup key with Apex side; `content_hash` in `chassis_contract.py` covers Python-to-Python hops only

**Provenance contract:**
`EnrichmentRunLogger` captures: `confidence`, `kb_content_hash`, `variation_count`, `consensus_threshold`, `uncertainty_score`, `inference_version`, `processing_time_ms`, `enrichment_payload`, `feature_vector`. This mirrors the Python `EnrichmentRun` model fields.

## 4. Critical Gaps for Agent Awareness

### Blocking (must resolve before any deployment action)

1. **Dual write-back, no coordination** — `EnrichmentWriteBack.cls` and `SalesforceClient` can both write to the same record. No idempotency key links them. Last-writer-wins with no cross-path audit trail.

2. **SOQL injection in `buildQuery()`** — `EnrichmentProfileResolver` interpolates keys from `Target_Fields__c` JSON directly into SOQL `SELECT` clause. Field names are user-controlled strings; `String.escapeSingleQuotes` only protects string values, not identifiers. Security Review blocker.

3. **Bind variable missing in `EnrichmentOnDemandController`** — `input.recordId` is embedded in SOQL string using `String.escapeSingleQuotes()` rather than a bind variable. Security Review blocker.

4. **`sfdx-project.json` has no `packageId`** — package is not yet registered against a Dev Hub. `sf package version create` will fail.

5. **Sync `httpx.post()` in async app** — `SalesforceClient.connect()` uses blocking HTTP in an async FastAPI process. Blocks event loop during OAuth2 token fetch.

### High (degrade correctness or observability)

6. **`Missing_Fields_Only__c` unimplemented** — field exists on `Enrichment_Profile__c` but `buildQuery()` uses a `Last_Run_Date__c` proxy; does not actually filter by whether target fields are blank.

7. **3 Python files not in `SOURCES.txt`** — `inference_unlock_scorer.py`, `confidence_tracker.py`, `convergence_config.py` are on filesystem but absent from package manifest. Will not ship in distributions.

8. **`mypy` excludes 2 engine files** — `field_classifier.py` and `inference_bridge_v2.py` skip type checking. Untyped modules in production engine path.

9. **Coverage floor = 60%** — pyproject.toml threshold is 25 points below L9 Contract 17 minimum (85%).
